En este vídeo, vamos a aclarar algunos conceptos que a veces nos pueden confundir. Empecemos por **MVC**, el Modelo-Vista-Controlador. Se trata de un patrón de arquitectura del software para la implementación de interfaces de usuario que divide la aplicación en tres partes interconectadas: el modelo de datos, la interfaz de usuario y la lógica de control. 

Luego tenemos **Spring MVC**, uno de los módulos del framework de Spring, que sigue el patrón que acabamos de mencionar. Podemos decir que Spring MVC integra el framework Spring y el patrón MVC. Conserva las características típicas de Spring, como la inyección de dependencias y la inversión de control. 

Respecto al patrón MVC, Spring MVC, en el **modelo**, contiene los datos de la aplicación, normalmente en beans de Java. Ya sabes, las clases esas con atributos privados, getters y setters. La **vista** es la responsable de mostrar los datos del modelo, generando el código HTML que se muestra en el navegador del cliente. Y el **controlador** tiene, cómo no, la lógica de negocio de la aplicación. 

Teniendo claro esto, ya podemos decir que **Spring Boot nos proporciona ciertas configuraciones por defecto** para poder usar Spring MVC en el desarrollo de aplicaciones web. Dicho esto, uno de los usos más comunes de Spring Boot es para crear aplicaciones web, y las aplicaciones web creadas con Spring Boot utilizan Spring MVC, para el que nos proporciona configuraciones por defecto. 

El starter `Template-engine` proporciona las herramientas necesarias para resolver las vistas, así como los ficheros estáticos, como puede ser el **index**, pero también ficheros **JavaScript** o **CSS**, o cualquier otra cosa que pongamos en la carpeta `static`. 

Aunque lo pondremos en práctica en breve, listemos los pasos a seguir para construir una aplicación web utilizando Spring Boot:

1. Empezamos creando un proyecto sobre el que trabajar, sobre el que definiremos las vistas, que quizá no hagamos nosotros.  
2. También definiremos los modelos que usaremos para proporcionar los datos a las vistas.  
3. Finalmente, construiremos un controller que se encargará de enlazar vista y modelo y pasárselo a la máquina de plantillas para su procesado final.
